# Code Governance

- 本文件用于补充代码编辑细则，不替代 AGENTS.md 中的工作流协议、授权边界和全局查证规则。
- 如存在缺失信息，先补齐相关包、工具、模块、发布源和维护方信息，不得凭命名或经验猜测实现方式。
- 修改前先确认目标属于仓库内实现、第三方依赖、框架能力、模板资产还是脚手架产物。
- 对仓库内实现，优先查看定义、引用、README、技能文档、模板说明和相邻设计文档，再决定修改点。
- 对第三方依赖，优先确认包名、命名空间、模块名、发布源以及官方文档或官方仓库；用户未提供官方链接时，优先使用包管理元数据、官方文档、官方 GitHub 仓库或官方 Gitee 镜像。
- 对第三方工具、框架或包，先查官方仓库、官方文档、官方发布页与官方博客；只有这些来源和仓库源码都无法回答问题时，才允许把反编译作为最后手段。
- 若存在多个候选仓库，只采用官方组织或官方文档、发布页指向的源码仓库。
- 只做必要修改，保持补丁最小、可审阅、与请求直接相关；避免借机清理无关代码。
- 不得引入未经验证的依赖、代码占位符、不完整实现或过时方案。
- 不得跳过官方资料查证流程，直接依据反编译结果、博客摘录或零散搜索结果实施修改。
- 修改时应提供最小必要上下文，保留可审阅性，保持标准化命名与适当错误处理。
- 新增、修改或重构代码和函数时，必须检查是否已有明确注释说明；若没有，必须补充。
- 修改复杂、关键或容易误解的逻辑时，补充或修正注释，说明职责、实现原因、关键前置条件和上下游关系；若旧注释失真，必须同步修正或删除。
- 若代码属于缺陷修复、兼容性修补、边界条件修正、异常路径修复、fallback、临时规避或补丁式调整，必须补充结构化注释说明修补原因。
- 上述修补类注释至少应说明：为什么这样修、修复了什么问题、适用边界或前置条件、后续在什么情况下可以移除或重构。
- 不得只写"修复 bug""兼容处理""临时方案"这类没有上下文的模糊注释。
- 下列场景必须优先补充详细注释：关键函数、公开 API、Activity/Fragment 入口与生命周期编排、依赖注入注册链路、工厂与模板链路、适配器与回调链路、异步流程、缓存一致性、兼容性分支、回退逻辑。
- 注释不得只复述代码表面动作，应说明这段代码或函数在整体流程中的职责、为何这样实现、依赖什么前置条件，以及会影响哪些上下游模块。
- 新增、修改或重构代码时，应在关键位置添加日志以观测运行过程，至少覆盖：入口开始、关键分支、第三方接口或模型调用、重试与降级、异常捕获、结果完成或失败返回。
- 全仓库通用日志规则：统一使用 `library/log` 模块的 `EasyLog` 作为日志入口（`EasyLog.print(msg)`、`EasyLog.print(tag, msg)`、`EasyLog.print(Throwable)`），不要在业务代码中直接调用 `android.util.Log` 或新增并行日志封装。
- 若未来需要更换日志组件，应优先修改 `library/log` 模块的桥接层，而不是在业务代码中分散替换日志实现。
- 普通流程使用 `EasyLog.print()`，调试过程使用 `DebugLog`，异常必须使用 `EasyLog.print(Throwable)`。
- 在高频热路径（如 `onBindViewHolder`、滑动回调）避免打印过多日志；Release 构建中 `LOG_ENABLE` 默认关闭，注意日志语句的性能影响。
